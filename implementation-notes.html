Arc Spirits ML-Only Bot Implementation Notes

Goal: implement the ML-only bot plan and make the documented testing criteria executable locally, in Browser checks, and on the SimForge GPU box.

Design Decisions

  • Start by making the acceptance criteria executable before deleting heuristic bot paths. This gives every later architecture change a reproducible local/GPU validation gate.
  • Use the current SvelteKit repo as the implementation source, then sync selected docs and runbooks to the remote arc-bot workspace on simforge1.
  • Use the Browser plugin for visible UI/dashboard acceptance, and Playwright where the flow is already scriptable.
  • Add arc-bot-v1 in src/lib/play/bots/contract.ts as the shared observation/action-id/decision contract. Live bot profile values normalize to neural.
  • Remove live-room dependency on strategic heuristic profiles. Missing bundled weights now fall back to uniform legal action selection through the same legal action surface.

Deviations

  • The historical heuristic planner is not deleted yet. It is quarantined as research/baseline scaffolding because multiple existing ML generation and meta tests still use it as a comparison field.
  • The Browser P0 test now drives setup and round advancement through authenticated play API calls, then uses real Browser contexts for reload/synchronization checks. This keeps the gate focused on multiplayer state, not fragile animated hit targets.

Tradeoffs

  • The repo currently has many uncommitted ML changes. The implementation works with that state rather than reverting it.
  • Long GPU runs are gated behind smoke/medium runs so expensive training does not hide broken local or contract behavior.
  • A true external bot repository should wait until the contract and run artifacts stabilize; for now the SimForge arc-bot workspace is a synced checkout, not an independent source of truth.
  • ML smoke data is isolated in ml/data_smoke so stale historical JSONL files with old observation/action dimensions cannot poison the acceptance gate.

Verification

  • npm run check: passed with 0 errors and existing warnings.
  • npm run test:bot:engine: passed engine/Vitest/legal-action parity gates.
  • npm run test:bot:browser: passed P0 multiplayer flow and mobile smoke.
  • npm run test:bot:ml-smoke: passed generation, one-epoch train, and eval.
  • npm run test:bot:az-smoke: passed local AlphaZero/meta smoke.

Open Questions

  • None blocking. The final heuristic deletion must preserve enough non-strategic random or uniform policy support for smoke tests.

Full-Screen Scene + Floating HUD Refactor Notes

Implemented the iPhone-landscape gameplay refactor so the scene owns the full viewport and the HUD floats above it, with shared navigation drag controls exposed by SpiritWorldBoard.

Design Decisions

  • The splat/main scene now sits under a full-screen transparent scene input layer, with traits, stage, leaderboard, top controls, and the phase rail layered above it.
  • SpiritWorldBoard remains responsible for carousel math, focused destination, card depth, and final snap/commit behavior. GameBoard2D only decides where empty-screen pointer input is captured.
  • Empty scene drags call the same beginDrag, moveDrag, and endDrag controller as direct card drags. Scroll snap is disabled only during the active drag so micro-movement visibly tugs the carousel, then restored for the settle.
  • MainStage now owns the single main-scene instruction row. Navigation renders Choose a Destination., location actions render Choose an Action., and individual stage branches no longer render their own page-level instruction headers.
  • The location action screen now has a separate scrollable card body under the shared stage instruction, so action cards can resize or overflow without pushing the prompt.
  • Direct card drags use pointer capture and the same no-native-pan carousel controller as blank scene drags; the controller imperatively disables scroll snap before the first movement frame, then restores snap for settle/commit.

Deviations

  • The original tapScene contract was removed after the product correction: empty-screen single taps should not change locations. Only drag/swipe gestures manipulate the carousel.
  • The active input layer is a component-local scene surface instead of document-level listeners, keeping real HUD controls isolated while blank scene areas remain draggable.
  • The top phase bar and navigation timer now show status/timing copy only. Main-scene instructions live exclusively in MainStage.
