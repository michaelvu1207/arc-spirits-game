# P30 plan & container-review implementation — review findings

**Summary:** The integrity architecture (role-separated Ed25519 custody, canonical-JSON signing, O_EXCL one-shot attempts, pinned Docker engine/daemon/image/config, exact-match receipt validators) is sound, and I found no outcome-exposure, forgery, or isolation hole. I verified the v13 fix (scheduler, launcher, and validator now all bind `ledger/requests/<mode>-fable-review.json`, with a test), the CUDA attempt-directory contract (empty-dir accepted; nonempty/non-dir/symlink rejected, all tested), and the socket-length contract (93/92 bytes under `/dev/shm/p/`, tested). I traced the gate-review launcher's receipt construction field-by-field against `validate_gate_review_receipt` and the analysis launcher against `_validate_analysis_review_payload`: they are currently consistent, including the positional `inputs[0]`=protocol / `inputs[2]`=source-contract assumption, which holds for both review modes.

However, applying this project's own rejection standard (v12 was rejected over a fetcher-robustness defect; v13 over a binding mismatch that would have fail-closed anyway), the following P1 gaps remain. All of them share one property: they sit in source-locked launchers that become **immutable once a seed is consumed**, so they can only be fixed before launch.

## P1 blockers

1. **No pre-reservation credential/API liveness probe.** `resolve_review_container` verifies the engine, daemon, image, and `claude --version` offline, but nothing validates the OAuth token or Anthropic API reachability before `reserve_remote_attempt`/`reserve_attempt` durably consumes the one-shot. An expired token (the analysis review may run weeks after minting), an API outage, or a 529 after reservation terminally closes the lane — for the analysis lane, that forfeits the entire 54-endpoint campaign with no recovery path (`analysis-review-attempt-consumed` is a dead end in `run_v35_p30_campaign.py:2328`). This is exactly the 135ae1 failure genre: an unrehearsed post-consumption runtime interaction. Fix: before reservation, run a minimal authenticated model call in an identically configured throwaway container (same fd-0 OAuth delivery, distinct name, dummy capsule), require exit 0 / empty stderr, and bind its evidence into the attempt.

2. **Everything after a captured `VERDICT: ACCEPT` is non-resumable.** In both launchers, `sign_with_vault`/`run_local_review_attester_signer` and `upload_review_evidence` follow `require_exact_accept`, and any failure (op-vault hiccup, scp timeout, SSH drop, a crash between reservation and signing) closes the consumed lane even though the review succeeded and its evidence is complete, hash-bound, and immutable on local disk. At the analysis stage this converts a 120-second `scp` timeout into total campaign forfeiture. A resume mode that revalidates the existing local capsule (attempt bytes match the consumed remote attempt, streams match their receipt hashes) and retries signing/upload is fully exactly-once-compatible — the bytes are already fixed. Relatedly, the local attester has no durable signing-attempt marker at all, unlike every remote role (plan §Phase 0 item 5), so a crash there leaves only ambiguity.

3. **Launcher receipt construction is never exercised by the validators in tests.** `test_v35_p30_phase0_gates.py` and `test_v35_p30_analysis_review_local.py` both hand-build receipt payloads that mirror the validators, rather than driving `launch()`'s actual construction (with ssh/docker/signing mocked) through `validate_gate_review_receipt`/`validate_analysis_review_receipt`. The gate lane additionally has no runtime pre-sign contract validation (the analysis lane validates before signing; `_sign_from_stdin` in the gate launcher signs whatever JSON is at `--unsigned` with no payload validation). v12 and v13 were both launcher↔validator drift caught only at review time; the next such drift is caught only *after* a consumed attempt, i.e. a fifth terminal instance. Fix: factor receipt construction into pure functions used by `launch()`, validate them in tests via the real validators for both modes, and add pre-sign validation (and the `read_secret_fd` pipe check) to the gate signer.

## P2 issues

- **Cross-host clock check post-consumption:** `_validate_analysis_review_payload` requires Mac-clock `startedAtUtc` > SimForge-issued `issuedAtUtc` of draft/manifest (`v35_p30_analysis_review.py:708`). Modest skew fails the receipt after consumption. Probe/record skew pre-reservation.
- **Gate launcher input hygiene:** `fetch` and `scheduler_status` in `run_v35_p30_gate_review_local.py` lack the analysis lane's `SAFE_REMOTE_PATH`/ledger-prefix and single-line-stdout checks, and gate inputs get no metric-free envelope check at all — a compromised remote scheduler can name arbitrary remote files (outcome-bearing or prompt-injecting) as reviewer inputs.
- **Timeouts are terminal, so size them generously:** 3600s for a high-effort review of a 595-receipt manifest, and 120s `transfer_timeout` also gating the remote scheduler call that late-campaign re-hashes the whole ledger chain, are both plausibly too tight.
- **Daemon default seccomp is `unconfined`** in the pinned daemon config; the per-container `seccomp=builtin` override is what actually enforces filtering (and the real-Docker smoke verifying `Seccomp: 2` is good). Consider fixing the daemon default; the pin currently freezes a weak host posture.
- **Metric-leak walk is a denylist** (`wins`, `winRate`, …) that misses e.g. `score`, `elo`, `vp*`; acceptable only as defense-in-depth behind the guardian's metric-free signing.

## Plan-level notes

- The 135ae1 incident record says `gameSeedsConsumed: true` (the 4,096-game CUDA determinism preflight), while the recovery policy demands "wholly fresh paired seeds" for restart. Presumably preflight seeds are disjoint from the paired training/evaluation schedule — make that explicit in the plan amendment.
- Five instances/reviews have now died pre-launch, each rerunning GPU-consuming Phase-0 preflights outside the 14-day ceiling's accounting; the plan should track cumulative cross-instance GPU spend.
- The 16-byte lifecycle-smoke stdout suggests a real end-to-end Claude call produced `VERDICT: ACCEPT\n`, but the plan text doesn't state that the smoke exercised the full argv (`--model fable --effort high --tools Read`) with fd-0 OAuth against the live API; make that an explicit, registered precondition (it overlaps with P1-1).

Because P1 items 1–3 sit in files that are frozen at source-lock time and each converts a routine transient failure into permanent forfeiture of an entire campaign instance — or, at the analysis stage, the entire campaign — they must be resolved before a fresh instance is authorized.
