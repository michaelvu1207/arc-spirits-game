## Remaining P1 launch blocker: RECEIPT_ONLY recovery is fully wired for the analysis kind

The plan (Phase 4) states: "Analysis deliberately has no recovery path… Any analyzer launch or sealing failure closes this campaign lane… The bounded PRE_CHILD/RECEIPT_ONLY state machine applies only to generation and evaluation work," and the frozen `recoveryPolicy` pins `eligibleKinds: ["generation", "evaluation-primary", "evaluation-replay"]`. The implementation contradicts this:

- `v35_p30_recovery.py` `ALLOWED_KINDS` includes `"analysis"` (and `"preflight"`), and `eligibleKinds` is validated only as a frozen literal in `validate_protocol` — it is never enforced at classification, draft, link, or seal time.
- `run_v35_p30_campaign.py:2029` routes the analysis execution through the generic `next_execution_action`, whose `ensure_recovery_execution_draft` will materialize a RECEIPT_ONLY draft for a completed-but-unsealed analysis and emit the guardian incident request with no kind check.
- `run_v35_p30_role.py` deliberately supports this: `_seal_recovery_receipt` (lines 978–988) and `_prepare_receipt_only_recovery_seal` (lines 1035–1044) contain explicit `kind == "analysis" → analysis-authorizer` verification branches whose only purpose is to seal an analysis-kind RECEIPT_ONLY recovery. The asymmetry is telling: the PRE_CHILD path correctly raises "PRE_CHILD recovery is forbidden for this authorization kind" for analysis, while the RECEIPT_ONLY path accommodates it end-to-end (scheduler → guardian incident → link → executor seal → completion).

This is not merely a documentation mismatch. The plan's stated rationale for giving analysis no recovery path is that once the analyzer has run, outcome blindness can no longer be proven from supervisor evidence. A RECEIPT_ONLY analysis recovery inserts multiple discretionary, human-mediated steps (guardian incident, link, key release, completion) *after* the outcome-bearing `analysis.json` exists on disk but before it is sealed and committed. That creates exactly the peek-then-decide window the plan closed: a party observing the unsealed result could choose between completing the recovery (committing a favorable result) and letting the lane close (burying an unfavorable one, then restarting under a fresh campaign ID), a selective-reporting channel over the primary outcome. Under the plan's rule — sealing failure closes the lane unconditionally — no such choice exists.

Required resolution before authorization: enforce the frozen `recoveryPolicy.eligibleKinds` at recovery time — raise on `kind == "analysis"` (and `"preflight"`) in `ensure_recovery_execution_draft`/`persist_receipt_only_recovery_draft`, remove the analysis-authorizer branches from the role runner's recovery seal paths, and have the scheduler treat any analysis draft/seal-attempt state as terminal lane closure — or, if RECEIPT_ONLY-for-analysis is genuinely intended, amend the plan and the frozen protocol first and re-review. Either way, plan and implementation must agree before this trust boundary is signed.
