The plan's cryptographic/isolation machinery is unusually thorough, and the statistical pre-registration (frozen power artifact, Holm, conjunction gates, honest sensitivity disclosure) is solid. The important risks are now mostly operational: the trust ceremony itself has caused all six instance closures to date, and the all-or-nothing failure policy interacts badly with the campaign's scale. Findings, roughly by severity:

**1. No reliability budget for ~541 child executions under an instance-fatal failure rule.** Phase 2–4 require 432 generation runs, 108 evaluations, and 1 analysis, sequentially over up to 14 days on one physical GPU. Any child-started failure without a sealable draft (driver reset, OOM, power blip, host death mid-run) permanently closes the instance and discards everything — the bounded recovery policy only covers `PRE_CHILD`/`RECEIPT_ONLY`. The plan never estimates the probability of completing 541 executions with zero such events. If per-child hard-failure probability is even 0.1%, completion probability is ~58%. Concrete improvement: add a pre-registered, outcome-blind replicate-quarantine rule (e.g., analysis proceeds with ≥16 of 18 matched replicates, power artifact recomputed for the reduced count) instead of instance-wide closure for isolated mid-child infrastructure failures.

**2. The same all-or-nothing rule applies to training NaNs.** A single non-finite checkpoint anywhere in 432 training runs closes the instance. That's the right call for determinism/integrity violations, but NaN loss is a stochastic training event, not a trust event. At minimum, justify this with the empirically observed NaN rate from V34 at these hyperparameters; better, fold it into the replicate-drop rule above.

**3. Integration bugs are only ever discovered at the real launch boundary.** All six closures (directory-exists contract, `/proc/driver/nvidia` submount, 110-byte socket path, macOS `security` helper, insufficient rehearsal, `scp` fetcher) were harness/integration defects, each costing a full restart. The v2 analyzer rehearsal fixes this for one component. Extend the same idea to a full miniature end-to-end dress rehearsal as a Phase-0 gate: 1–2 synthetic replicates × 1 generation through the real trainer → evaluator → guardian → analysis path under a disposable child campaign ID and fixture seeds. It's cheap relative to another closure and would have caught most of the six incidents.

**4. A transient Fable API failure in the Phase-4 review permanently closes the analysis lane after the full campaign spend.** The liveness probe and resumable postprocessing mitigate but don't eliminate this: an API 500 mid-attempt is "model/review failure … without retry." Since the review capsule is metric-free by construction, a verdict-free failed attempt leaks nothing; the no-retry rule is stricter than the threat model requires. Suggest a small committed attempt budget (like the pre-launch reviews already allow) for attempts that provably produced no verdict, with each disposition sealed.

**5. Mid-campaign ceiling breach behavior is unspecified.** Phase 1 makes exceeding the projected 14-day ceiling a no-go at preflight, and generation boundaries recompute projections, but the plan doesn't say what happens if actual runtime crosses the ceiling mid-campaign despite passing projections (later generations may run longer as policies improve and games go deeper). Make the breach action explicit and outcome-blind.

**6. No cross-instance stopping rule.** The 14-day ceiling and spend accounting are per-instance; the cumulative-spend artifact records but doesn't decide. Six instances have closed before generation 1. Freeze a cumulative cross-instance budget or attempt-count threshold that forces a design retrospective rather than an automatic seventh identical restart — especially since a closure on day 13 of Phase 2 would burn a full campaign's compute.

Smaller points:

- **Dose confound reporting:** the sealed realized-dose manipulation checks (rounds 1–8/9–18/19–30) are recorded but the analysis output isn't committed to reporting them. Pre-register their inclusion so the joint "schedule-and-dose package" claim is interpretable, since the plan already concedes scheduling can't be isolated.
- **Michael gate calibration:** 28/40 sounds like 70%, but at a true 70% win rate the pass probability is only ~50%; the gate effectively demands ~78–80% true skill for reliable passage. Worth stating the assumed true rate and implied power, as done for the primary screen.
- **Marginal power:** the frozen joint-power bound (0.80907) clears 0.80 by 0.9%; any post-freeze change to the endpoint set or gates invalidates it. Already implicitly handled by freezing, but worth flagging as zero slack.
- **Document structure:** the six incident amendments now dominate the file and bury the normative contract. Move incident history to an appendix or separate log; reviewers of instance N+1 need the current contract, not the archaeology, and drift between amendment prose and the actual frozen JSON is a growing risk (the plan itself admits the summary "is not a substitute" — keep that surface small).

The single highest-leverage change is #3 (full end-to-end dress rehearsal), followed by #1 (replicate-level quarantine): together they attack both the dominant historical failure mode and the dominant prospective one.
