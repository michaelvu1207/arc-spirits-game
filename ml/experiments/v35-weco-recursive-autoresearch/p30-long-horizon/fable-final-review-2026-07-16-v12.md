## Remaining launch blocker

**P1 — the Phase-0/full-campaign gate-review launcher cannot download its inputs, and it is source-locked.** In `ml/run_v35_p30_gate_review_local.py`, `fetch()` (lines 125–142) first creates the local target with `os.open(target, O_WRONLY|O_CREAT|O_EXCL, 0o400)` and closes it, then invokes `scp` to write into that same path. A mode-0400 file owned by a non-root user cannot be reopened for writing, so every scp into the pre-created target fails with EACCES, deterministically, on the first input of every gate review. The sibling analysis launcher does this correctly (`fetch_remote_file` lets scp create the file and only then `os.chmod(target, 0o400)`), which confirms the gate-review variant is a defect rather than an intended pattern. Consequences: the keyless scheduler cannot pass `phase0-readiness` (and later `full-campaign-authorization`) without a receipt this launcher produces, so a launched instance would run the expensive two-server 4,096-game CUDA determinism preflight and then be permanently stuck at `awaiting-phase0-runtime-fable-review`; because the launcher is in the frozen source registry and lock, the one-line fix forces a full instance closure and regeneration under the plan's own amendment rules. This must be fixed (and covered by a regression test, as the last three incident fixes were) before any source lock is signed. A secondary provenance note in the same lane: unlike the analysis review receipt, the gate-review receipt does not bind `launcherSha256`, so the broken sanctioned path could only be worked around by unattested manual reproduction — exactly what the local-only provenance requirement exists to prevent.
