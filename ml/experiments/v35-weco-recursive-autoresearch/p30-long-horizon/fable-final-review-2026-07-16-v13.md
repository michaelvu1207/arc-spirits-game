## Remaining launch blocker

**P1 — the gate-review receipt can never validate: the `request` binding is checked against a file no component ever creates, at a path the receipt does not record.** `validate_gate_review_receipt` requires `receipt["request"] == binding(paths["request"])` (`v35_p30_phase0.py:185`), where `paths["request"]` is `ledger/reviews/<mode>/review-request.json` (`v35_p30_phase0.py:73`). But the scheduler emits the review request into the ledger's `requests/` directory — `_review_required_status` binds `reviewRequest` to `_request_path(...)` = `ledger/requests/<mode>-fable-review.json` (`run_v35_p30_campaign.py:312-314`, `1324-1325`) — and the launcher copies that binding into the receipt verbatim (`run_v35_p30_gate_review_local.py:410`). Nothing in the entire closure writes `reviews/<mode>/review-request.json`: the launcher uploads only stdout, stderr, and the signed receipt; `reserve_attempt` writes only `attempt.json`; the role runner writes requests only under `requests/`. Consequences are deterministic and doubly fatal: `binding(paths["request"])` raises `FileNotFoundError` on a nonexistent file, and even if the file were materialized, the receipt's recorded path string (`requests/…`) can never equal the expected path (`reviews/…/review-request.json`). Every consumer — `_phase0_next`, `require_phase0_readiness`, `_attest_phase0_readiness`, `_authorize_full_campaign`, `require_full_campaign_authorization` — calls this validator, so `phase0-readiness` and `full-campaign-authorization` can never be signed. By that point the one-shot review-attempt token is already consumed and the expensive 4,096-game two-server CUDA determinism preflight has already run, so a launched instance would be permanently stuck at exactly the failure class v12's P1 described, one step later in the pipeline. The unit test masks this because it constructs the receipt payload directly from `binding(paths["request"])` rather than exercising the launcher-produced receipt shape (`test_v35_p30_phase0_gates.py:209`); the fix needs a regression test that binds the two sides together, as the last four incident fixes each received. Both files are source-locked, so this must be repaired before any source lock or campaign ID is generated.
